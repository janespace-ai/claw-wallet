## Context

当前文档仍描述 Phase 1 的单组件架构（`ClawWallet` SDK 内嵌在 Agent 进程内，私钥本地加密存储）。实际项目已演进至 Phase 2 三组件架构：

- **Agent** (TypeScript)：零秘密，通过 JSON-RPC/IPC 发起请求
- **Desktop Wallet** (Electron)：持有加密私钥、签名引擎、安全监控
- **Go Relay Server** (Hertz)：无状态 WebSocket 转发，E2EE 加密消息中继

同时已实现：自动配对与重连（持久化通信密钥对、确定性 pairId）、三级重连验证（公钥连续性 / machineId / IP 策略）、Relay 侧 IP 绑定与连接限流、手动重配对兜底。

文档需要全面刷新以反映真实系统状态。

## Goals / Non-Goals

**Goals:**
- 让新用户 10 分钟内理解系统运作方式（用户交互流程前置）
- 安全架构作为核心卖点突出展示（E2EE、三级验证、密钥隔离）
- 所有 9 种语言的 README 同步更新
- 辅助文档（PHASE2-ELECTRON-WALLET, AGENT_VERIFICATION）反映最新方案
- 保持文档间的一致性

**Non-Goals:**
- 不修改任何源代码
- 不引入新功能或 API
- 不编写 API reference 文档（保持在工具表格级别）
- 不翻译为新语种

## Decisions

### 1. README 章节重新排序

**决定**：采用以下章节顺序——项目介绍 → 用户交互流程（首次配对 + 日常使用）→ 安全架构（重点）→ 功能列表 → 快速开始 → 工具列表 → 配置 → 架构 → 开发

**理由**：用户最先关心的是"这东西怎么用"和"安全不安全"，而不是代码结构。将交互流程放在架构图之后、安全设计之前，让用户先建立整体认知。

**替代方案**：保持原有顺序（功能 → 场景 → 快速开始 → 安全）——放弃，因为旧顺序是 SDK 模式的逻辑，不适用于三组件架构。

### 2. 安全模型章节结构

**决定**：安全模型分为两大部分——
- Part A: 通信安全（E2EE、自动配对/重连、三级验证、Relay 侧保护）
- Part B: 密钥安全（Keystore V3 加密、内存安全、策略引擎、输入验证、文件系统安全）

**理由**：Phase 2 的核心差异是通信安全，应放在第一位。密钥安全是从 Phase 1 继承的成熟方案，放在第二部分。

### 3. 架构图更新

**决定**：主架构图展示三组件 + E2EE 通道，取代旧的单进程 Tool→Policy→Keystore 图。保留旧图作为 Desktop 内部细节的子图。

### 4. 多语言翻译策略

**决定**：英文版作为主版本完成后，其他 8 种语言直接按英文重写，保持完全结构一致。

### 5. PHASE2-MOBILE-SIGNER.md 处理

**决定**：文件头部标注为 "Phase 3 远景设计"，正文内容保持不变，增加一段说明当前 Phase 2 已使用 Desktop Wallet（而非 Mobile）作为签名端。

## Risks / Trade-offs

- **翻译质量**：AI 翻译可能不如母语者自然 → 优先保证英文和中文质量，其他语言标注"机器翻译，欢迎贡献"
- **文档过长**：安全细节过多可能吓跑新用户 → 使用折叠/摘要，安全模型分层级展示（概述 + 详细）
- **文档维护成本**：9 个 README 文件同步更新成本高 → 建立英文主版本 → 翻译版本的模式，后续可考虑自动化
