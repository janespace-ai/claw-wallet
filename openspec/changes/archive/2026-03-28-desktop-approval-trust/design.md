# Desktop 审批 / 信任 / 联系人——设计结论（Explore 汇总）

本文档固化与产品/架构相关的**已定方向**，供实现与评审对齐。细节（表结构、消息 JSON schema）在开发阶段补足。

---

## 1. 信任边界：谁算「多少钱」、谁决定是否签名

| 决策点 | 权威 |
|--------|------|
| 代币美元估值（用于限额、是否可静默签） | **Desktop**：`PriceService` + 链上金额 |
| Agent 传的「美元」或 `amountUsd` 类字段 | **不作为**签名闸门的真值；仅可作展示或弃用/改名 |
| 无法获取价格 | **禁止自动签**，必须人工审批 |

**推论**：修复「字段误导 + Desktop 自用现价」后，原先「白名单 + 错误低价 → 超额仍自动签」一类问题在机制上被消除；**白名单（信任）仅在与「有可信 USD 且在限额内」联立时**才允许静默签（产品规则实现时保持此组合）。

---

## 2. Agent 载荷与命名

- 应对 Agent → Desktop 的字段做**消歧义**（例如明确 `amount_token` / `value_wei` 等），**禁止**在未验证情况下把代币数量标为 `Usd`。
- Desktop 在 `handleSignRequest`（或等价入口）根据 **token + value + decimals**（及既有 `PriceService`）**自行计算** `estimatedUsdForPolicy`。

---

## 3. 白名单（信任地址）与 Agent `policy.whitelist`

- **信任地址仅存储于 Desktop（SQLite）**，且**添加必须经过 Desktop 人工确认**。
- **移除** Agent 侧「私自改白名单即生效」的路径（移除 `wallet_policy` 中 `add_to_whitelist` 的本地生效语义，或整段能力迁移到「仅向 Desktop 申请」）。
- **监督/自主模式**若在 Agent 侧保留，其语义应收缩为「业务层是否允许发起签名请求」，**不**再承担「谁可信」的最终定义；最终以 Desktop 信任列表与限额为准。

---

## 4. 联系人：双边存储，Desktop 权威

- **Agent** 与 **Desktop** 均可存联系人；**不要求**两边实时一致。
- **解析收款地址、合并展示、冲突时**：**以 Desktop 为准**。
- **Agent 发起的联系人增删**：经 **Relay → Desktop 主进程**；**不新建** Agent↔Desktop 并行通道，**不**采用双方之间的 IPC；Desktop 内持久化与界面刷新沿用应用现有主进程 / 前端协作方式（本变更不要求引入新的跨端 IPC 方案）。
- **添加信任地址**：必须经过人工确认（可与签名审批合一，或单独队列）。
- **删除信任地址**：不需要 Agent 侧人工确认（Desktop 执行即可）。

---

## 5. 审批 UI：勾选「加入信任」

- 用户在审批界面可勾选将 **当前交易的 `to` 地址** 加入信任列表。
- **已定（方案 B）**：**不在勾选当下写入**信任库；待 **签名完成且交易广播成功并得到成功回执**（具体以产品与链上策略为准，如 receipt `success`）**之后**，再将该地址写入 Desktop 信任存储。若交易最终失败或回执为失败，则**不写**入。实现上可将「待加白」与 `requestId`/`txHash` 暂存，在成功回执路径落库。

---

## 6. 跨端通信（Relay 增协议）

- **不新建通信渠道**：延续现有 **Agent ↔ Relay ↔ Desktop**（HTTP/WebSocket + 加密载荷）转发；联系人、信任相关能力通过 **新增加密请求类型 / method** 扩展协议即可。
- **Agent ↔ Desktop 不采用 IPC**（无本机管道、无并行直连通道与 Relay 并列）。Relay 服务端仅按需透传，**不以本变更为由另开一条消息总线**。

---

## 7. 持久化

- **信任地址**与 **Desktop 权威联系人**数据写入 **Desktop SQLite**（与现有 `wallet.db`/迁移策略对齐，避免 `allowance.json` 中 `addressWhitelist` 与 DB **长期双源**——实现阶段应合并或迁移为单一事实源）。

---

## 8. 与既有问题的对应

| 现象 | 根因方向 | 本变更应对 |
|------|----------|------------|
| 白名单地址仍静默签超额 | 多用错误的「美元」参与 `checkBudget` | Desktop 自算 USD + 无价必审 |
| Agent 改白名单即可一直转 | 信任源在 Agent | 信任仅 Desktop + 人工 |
| 缺少统一管理入口 | 无 Tab / 无 DB | SQLite + 信任 Tab + 审批勾选 |

---

## 9. 非目标（本阶段 Explore）

- Relay 服务端大改（除非为透传新 method 所必需）。
- 多账户、硬件密钥。
- 精确法律/合规文案。
